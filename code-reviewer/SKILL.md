# code-reviewer

该技能指导代理对本地开发和远程拉取请求进行专业且彻底的代码审查。

## 工作流程

### 1. 确定审查目标
*   **远程 PR**：如果用户提供了 PR 编号或 URL（例如，“审查 PR #123”），则以该远程 PR 为目标。
*   **本地更改**：如果没有提及特定的 PR，或者用户要求“审查我的更改”，则以当前本地文件系统状态（已暂存和未暂存的更改）为目标。

### 2. 准备工作

#### 对于远程 PR：
1.  **检出**：使用 GitHub CLI 检出 PR。
    ```bash
    gh pr checkout <PR_NUMBER>
    ```
2.  **预检**：执行项目的标准验证套件，以便尽早发现自动化失败（注意：可选择跳过）。
    ```bash
    npm run preflight
    ```
3.  **上下文**：阅读 PR 描述和任何现有评论，以了解目标和历史记录。

#### 对于本地更改：
1.  **识别更改**：
    *   检查状态：`git status`
    *   阅读差异：`git diff`（工作区）和/或 `git diff --staged`（已暂存）。
2.  **预检（可选）**：如果更改量很大，请询问用户是否希望在审查前运行 `npm run preflight`。

### 3. 深入分析
根据以下支柱分析代码更改：

*   **正确性**：代码是否在没有错误或逻辑错误的情况下实现了其既定目的？
*   **可维护性**：代码是否干净、结构良好，并且未来易于理解和修改？考虑代码清晰度、模块化以及是否遵循既定的设计模式等因素。
*   **可读性**：代码是否在必要时有良好的注释，并且根据我们项目的编码风格指南保持一致的格式？
*   **效率**：更改是否引入了任何明显的性能瓶颈或资源低效？
*   **安全性**：是否存在任何潜在的安全漏洞或不安全的编码实践？
*   **边缘情况和错误处理**：代码是否适当地处理了边缘情况和潜在错误？
*   **可测试性**：新代码或修改后的代码是否得到了足够的测试覆盖（即使预检检查通过）？建议增加能提高覆盖率或稳健性的额外测试用例。

### 4. 提供反馈

#### 结构
*   **摘要**：审查的高层概述。
*   **发现**：
    *   **严重问题**：错误、安全问题或破坏性更改。
    *   **改进建议**：关于提高代码质量或性能的建议。
    *   **吹毛求疵**：格式或次要样式问题（可选）。
*   **结论**：明确的建议（批准 / 请求更改）。

#### 语气
*   具有建设性、专业且友好。
*   解释*为什么*需要进行更改。
*   对于批准，要认可贡献的具体价值。

### 5. 清理（仅限远程 PR）
*   审查后，询问用户是否希望切换回默认分支（例如 `main` 或 `master`）。
